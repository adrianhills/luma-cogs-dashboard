How to read this — a 5-minute info session

1 · See the cost

The top half just shows the bill — by surface, model, generation, and plan. Nothing's been changed yet. You can't fix what you can't see, so I start here.

2 · See the impact

Own model vs. third-party API

Our own model (Ray) is cheap to run — its training cost only shows up in the fully-loaded lens. Veo, Kling, and Seedance are other companies' models we call over an API; that fee is COGS too.

What this says: our own model makes a healthy ~56%. The third-party models make almost nothing (~8–11%) — we're just reselling someone else's API, and there's no way to engineer that fee down. They're only ~18% of generations but ~51% of the bill. So: move more volume onto Ray as it gets better, and price API clips to cover their cost. Flip to fully-loaded and Ray's lead shrinks — that's where its training cost finally lands, and the API models carry none.

How I'd run this — operating principles

Lightweight, living, and built with the teams who own the data — not another process to feed
Lightweight, not heavy process

One dashboard and one weekly review. Make things visible first; add formal controls only where visibility isn't enough. No chargeback paperwork, no FinOps committee, no Gantt chart to feed.

A living model, not a rebuild each release

When a new model ships, I update the rate card and re-baseline only the parts it touches — not the whole thing. The structure stays; the numbers refresh. It's a 30-minute job, not a project.

Partner, don't police

Build on the workflows the data owners already run: Finance / FP&A for the cost rate card, Sales for enterprise contracts and margin, and Analytics & Eng for the telemetry, tagging, and pipeline. My job is to connect them, not to start a new silo.

Educate so the team self-serves

Make the numbers easy to read. Once people can see what a 4K clip or an API route really costs, they make the call themselves — and I don't have to be in the room for every decision.
